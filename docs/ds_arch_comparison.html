Direction-Speed Architecture

Direction-Speed：从 SAC 探索语义到动作分层

本页按 approach.md 的顺序组织：先交代 action space、SAC actor/critic 和 explore/exploit， 再说明 Cartesian 高斯为什么会诱导不稳定的方向视锥，随后引出 direction × speed 分层， 最后对比 stereographic、spherical、post-hoc 与 bijector 的实现层级。

1. 背景：SAC 随机策略 actor 输出 μ、σ；critic 学 Q(s,aπ)；σ 承担探索语义。
2. 问题：XYZ 高斯耦合 固定 σ 的 95% 动作置信椭球会随 ||μ|| 诱导不同方向视锥。
3. 方法：方向 × 速度 把 3D 位移拆成 unit direction + speed，让随机性沿任务语义流动。
4. 实现层级：概率参数化 stereographic/spherical 做可逆分布；post-hoc 作为外部表示消融。

1. 背景：动作空间与 SAC

先看随机策略在什么坐标里探索

OGBench 5D 动作里，前三维是末端位置增量 dx, dy, dz，通常是 expressed in world frame 的 EE position delta；后两维是 yaw 和 gripper。SAC 在这个动作空间上学习随机策略。

1
Action space 先决定方向语义 Delta World 的方向是全局坐标方向；Delta EE 的方向是末端局部方向。本文讨论的是前三维位移向量上的方向与速度。
2
Critic 拟合 TD target Q(s,aπ) 评估训练用 policy action，训练目标是 Bellman backup 形成的一步 TD target；采样的下一步动作来自当前 actor。
3
Actor 输出 μ 和 σ μ 表示动作分布中心，偏 exploit；σ 表示采样尺度，偏 explore。问题在于 σ 的坐标语义是否可解释。
4
Explore / exploit 要能被解释 如果 σ 只是在 XYZ 坐标轴上变大，就很难说它增加的是方向探索、速度探索，还是二者的混合。

2. 问题预告：XYZ 表示不等于任务语义

问题不是维度，而是坐标系

直接用 TanhNormal 在 dx,dy,dz 上采样可以运行，但三维 Cartesian 坐标不是任务真正关心的语义坐标。位移动作同时包含方向和速度。

1. 方向和速度纠缠 (dx,dy,dz) 同时决定方向 v/||v|| 和速度 ||v||，一次噪声会同时改变两个语义。
2. σ 先形成动作置信椭球 Cartesian 高斯的 σx,σy,σz 定义的是固定置信水平下的动作采样椭球，而不是直接定义方向探索。
3. μ 的长度会改变方向探索 同样置信水平、同样体积的蓝色椭球，离原点越近，从原点看过去张开的方向视锥越大。
4. chunk 会重复这个问题 H=5 时每一步都有一组三维位移；方向、速度和标量动作的边界更需要清楚。

2.1 固定 σ 下，μ 长度如何改变方向视锥

Three.js 交互视锥

蓝色球/椭球表示 Cartesian 高斯在 [dx,dy,dz] 空间里的固定置信水平采样区域，例如 95% 概率质量对应的一层等概率置信椭球；红色锥体表示从原点看过去，这个局部采样区域诱导出的方向探索范围。 保持 σ 不变时，只有 l=||μ|| 变小，方向角也会被放大。

这里的"95% 置信椭球"指联合置信区域（Mahalanobis 距离 ≤ χ²_D(p)，保证总概率质量 = 95%），不是"每维 95% 区间拼成的长方体"——后者只包含 0.95³ ≈ 85.7% 的联合质量。两者形状、含义都不同，不能混用。

视锥参数

θ ≈ asin(σ / l)

这里把均值方向固定在 x 轴，只改变均值长度和探索半径，用来隔离说明方向噪声为何会随动作幅度变化。

l 越小，蓝色置信椭球越接近原点，红色方向视锥越容易张开。
σ 表示 Cartesian 三维高斯的局部采样尺度；在 DS 中，方向 σ 和速度 σ 会被分开解释。
11.5° 方向半角 θ。越大表示同样采样噪声会造成更剧烈的方向变化。
0.20 σ / l。这个比值越高，Cartesian 探索越不像“沿原方向微调”。

这正是 direction-speed 分层的动机：让方向维度的 σ 直接控制方向探索视锥，让速度维度的 σ 只影响速度探索， 不再让 ||μ|| 隐式改变方向探索强度。

4. 两类概率化参数化与具体 mode

入口统一为 --ds_mode

这里对应 approach 的参数化部分：spherical 和 stereographic 是 actor 分布内部的可逆 DS bijector；post-hoc 来自 posthoc_direction_speed.py，主要作为模型外部表示消融。

先分清三种 action

从 actor 采样到 env.step

下面的差别只回答一个问题：SAC 训练看到的 policy action 和环境原生 env action 是不是同一个变量。 Baseline 中二者相同；pure bijector 中也相同；但 bijector 仍然可以和 post-hoc/adapter 组合，此时还要看后面的执行层 g。

z / ξ latent value actor 从 Gaussian 采样出的未 squash 值。Baseline 常写 z，bijector 常写 ξ。
aπ policy action actor 暴露给 SAC loss、log_prob 和 critic 的动作。Baseline 中是 tanh(z)，bijector 中是 f(ξ)；post-hoc 中是 6D 的 â，不是 compose 后的 a_env。
a_env env action 环境原生动作，也就是 dataset / replay buffer / env.step 真正使用的 5D 动作。
Baseline aπ = a_env = tanh(z)
Bijector aπ = f(ξ); 仅当 g = identity 时 a_env = aπ
Post-hoc aπ = a_hat(6D), a_env = compose(aπ)(5D)
Bijector + Post-hoc aπ = f(ξ), a_env = compose(aπ)

实现方式

  • actor 输出 D 维 TanhNormal；OGBench 是 [dx,dy,dz,dyaw,grip]。
  • critic、dataset、env 都使用同一个 raw 5D action。
  • 每个维度独立建模，三维位移的方向和速度没有显式结构。
严格 log_prob XYZ 轴向探索 baseline

数据流

Normal(μ, σ)
  -> raw sample z
  -> tanh
  -> aπ = [dx, dy, dz, dyaw, grip]
  -> a_env = aπ

Q(s,aπ) 和 log π(aπ|s)
都对应 tanh 后的 policy action。
此时也可写成 Q(s,a_env)。

实现方式

  • decompose: xyz -> unit_direction + speed_coord，其余维度原样保留。
  • speed_coord = 2 * (||xyz|| / R(u)) - 1，其中 R(u)=1/max(|u_i|)。
  • compose: 先 normalize direction，再用 ρ=(speed_coord+1)/2 还原 xyz=ρR(u)u。
D+1 表示 非可逆 FQL 可用

核心映射

r = ||xyz||
u = xyz / max(r, eps)
R(u) = 1 / max(abs(u))
speed_coord = 2 * (r / R(u)) - 1

ρ = (speed_coord + 1) / 2
xyz = normalize(u) * R(u) * ρ

a_env = clip([xyz, other], -1, 1)

Q(s,aπ) 和 log π(aπ|s)
内部都用 decomposed action a_hat。

实现方式

  • actor base distribution 仍是 D 维 Gaussian，但前三维 latent 解释为 theta_raw, phi_raw, speed_raw。
  • bounded sigmoid 将 theta 映射到 (0,π)，phi 映射到 (0,2π)，speed 映射成沿 cube radial bound R(u) 的归一化比例。
  • log-det 包含球坐标体积项 r² sin(theta) 和三条 sigmoid 的导数。
可逆 bijector Jacobian-corrected 极点附近更敏感

核心映射

ξ = [theta_raw, phi_raw, speed_raw, ...]
theta = π * sigmoid(theta_raw)
phi = 2π * sigmoid(phi_raw)
direction = [sin(theta)cos(phi), sin(theta)sin(phi), cos(theta)]
R = 1 / max(abs(direction))
r = R * sigmoid(speed_raw)

x = r sin(theta) cos(phi)
y = r sin(theta) sin(phi)
z = r cos(theta)

aπ = bijector.forward(ξ)
a_env = g(aπ)  # pure bijector: g = identity

实现方式

  • 前三维 latent 是 plane_x, plane_y, speed_raw，先在平面采样，再 inverse stereographic 到单位球。
  • 方向映射没有球坐标角度极点，实验中作为 RLPD/SAC 的 DS 主线。
  • log-det 使用球极投影面积项 4 / (1 + rho²)²，再乘上 r²、R(u) 和 speed sigmoid 导数。
推荐主线 可逆 bijector 方差更稳

核心映射

ξ = [px, py, speed_raw, ...]
rho2 = px² + py²
direction = [
  2px / (1 + rho2),
  2py / (1 + rho2),
  (1 - rho2) / (1 + rho2)
]

speed = R(direction) * sigmoid(speed_raw)
action_xyz = speed * direction
aπ = bijector.forward(ξ)
a_env = g(aπ)  # pure bijector: g = identity

5. Post-hoc 与 Bijector：转换发生在哪一层

先看数据流，再看实现

理解 3D 位移语义后，再区分两个可以组合的层级：post-hoc 在模型外改动作表示或执行转换，bijector 在 actor 分布内部改采样坐标。pure bijector 可以直接输出 env action；bijector 后也可以继续接 compose。

Overview

总览两个层级：post-hoc/adapter 决定 aπ 如何变成 a_env；bijector 决定 latent ξ 如何变成可计算 log_prob 的 aπ。

关键点：post-hoc 的 6D 是模型外部显式动作表示。它工程上直接，但 D+1 → D 不是 bijection，所以 SAC/RLPD 的 log_prob 不是环境动作的严格概率密度。这里 critic 的 a 也是 6D decomposed action； 它实际学习的是 Q_env(s, compose(a_6d)) 的近似语义。

关键点：bijector 内部的 “latent action” 是 ξ，forward 后得到的是 aπ。SAC 内部对齐的是 Q(s,aπ) 和 log π(aπ|s)。只有 pure bijector 且执行层 g 为 identity 时， aπ 才等于 a_env，此时也可以写 Q(s,a_env)； 如果后面还接 post-hoc compose，则 a_env = compose(aπ)。
