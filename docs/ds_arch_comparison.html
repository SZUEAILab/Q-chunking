Direction-Speed Architecture

Direction-Speed：先解决 XYZ 表示，再选择实现层级

本页按一条线往下读：先说明 raw dx,dy,dz 为什么不够，再用 3D 场景解释 direction × speed 的动作语义，随后对比 post-hoc 与 bijector 的数据流，最后落到 --ds_mode 的具体实现。

1. 问题：XYZ 坐标 独立高斯把 dx/dy/dz 当成三条轴采样，方向和速度被混在一起。
2. 语义：方向 × 速度 把 3D 位移拆成 unit direction + speed，让随机性沿任务语义流动。
3. 架构：外部 vs 内部 post-hoc 在模型外做坐标转换；bijector 在 actor 分布内部做可逆变换。
4. 实现：四种 mode 按 none、post-hoc、spherical、stereographic 逐一确认维度、映射和 log_prob。

为什么 raw XYZ 表示不理想

问题不是维度，而是坐标系

OGBench 5D 动作里，前三维是空间位移 dx, dy, dz。直接用 TanhNormal 在 XYZ 上采样可以运行，但它把 “往哪里走” 和 “走多快” 交给同一组三维独立随机变量表达。

1
方向和速度纠缠 (0.1, 0.1, 0.1) 同时改变方向和模长。actor 想提高速度时也会改变方向，想微调方向时也会改变步长。
2
探索噪声按坐标轴注入 独立 Gaussian 更自然地产生 x/y/z 轴向扰动，而不是沿 “目标方向” 改变角度、沿 “速度” 改变半径。
3
tanh 边界压缩球体几何 XYZ 每维各自 squash 到 [-1,1]，动作空间更像 cube；真实位移语义更接近 direction on sphere + radius。
4
chunk 后问题被重复 H 次 H=5 时不是一个 3D 问题，而是 5 个独立位移组。每一步都需要清楚地区分方向、速度、标量动作。

优化口径

从 raw axes 到 task coordinates

DS 的目标不是增加外部噪声，而是把 actor 的随机策略放进更匹配任务的坐标系；探索仍来自 actor 自己。

1. 只分解真正的 3D 组 OGBench 使用 [dx,dy,dz] 做 direction-speed；dyaw 和 gripper 保持普通 tanh scalar。
2. RLPD/SAC 用可逆 bijector spherical/stereographic 让 sample() 返回 env action，log_prob(action) 自动包含 inverse 和 log-det。
3. post-hoc 只作为表示消融 D+1 多对一，不能给 SAC/RLPD 严格环境动作密度；适合 FQL 或 approximate ablation。
4. chunk 按步复制 group H=5,D=5 时是 5 份 DS group + scalar group，actor/critic 都看到 HD=25。

先看数据流，再看实现

理解 3D 位移语义后，再区分两条工程路径：post-hoc 在模型外改动作表示，bijector 在 actor 分布内部改采样坐标。

Post-hoc DS

模型外部 · D+1
Dataset

decompose5D → 6D
6D batchdir, log-speed, yaw, grip

actor输出 6D decomposed action
6D output模型并不直接输出 env action
Critic
Q(s, a6d)a 是 decomposed 6D
Q_env via compose真实环境动作是 compose(a)
Execution
compose6D → 5D，多对一
Env receives 5D真实执行动作

关键点：post-hoc 的 6D 是模型外部显式动作表示。它工程上直接，但 D+1 → D 不是 bijection，所以 SAC/RLPD 的 log_prob 不是环境动作的严格概率密度。这里 critic 的 a 也是 6D decomposed action； 它实际学习的是 Q_env(s, compose(a_6d))。

Bijector DS

Actor 内部 · D
Dataset
5D raw不做 decompose
critic batcha 是 5D env action
Actor
MLP params5D Gaussian μ, σ
latent 5DGaussian-space action
bijectorforward / inverse / log-det
Critic
Q(s, a5d)a 是 bijector output
same actionQ 和 log π 对同一个 env action
Execution
output 5Dstructured env action
Env receives 5D不再 compose

关键点：bijector 内部的 “raw/latent action” 不是 post-hoc 的 6D decomposed action。 它仍是 5D，只存在于 actor 分布内部；forward 后的 5D output action 直接传给环境。 这里 critic 的 a 是这个 output 5D env action，因此 Q(s,a) 和 log π(a|s) 对齐到同一个动作变量。

每种 DS 的具体实现

入口统一为 --ds_mode

下面按代码里的真实路径描述：post-hoc 来自 posthoc_direction_speed.py，bijector 来自 rlpd_distributions/direction_speed.py。

实现方式

  • actor 输出 D 维 TanhNormal；OGBench 是 [dx,dy,dz,dyaw,grip]。
  • critic、dataset、env 都使用同一个 raw 5D action。
  • 每个维度独立建模，三维位移的方向和速度没有显式结构。
严格 log_prob XYZ 轴向探索 baseline

数据流

Normal(μ, σ)
  -> tanh
  -> [dx, dy, dz, dyaw, grip]

Q(s, a) 和 log π(a|s)
都对应 raw env action。

实现方式

  • decompose: xyz -> unit_direction + log_speed，其余维度原样保留。
  • OGBench 5D 进入模型前变成 6D = [ux,uy,uz,log_r,dyaw,grip]。
  • compose: 先 normalize direction，再用 exp(log_speed) 还原 xyz，最后 clip 到 [-1,1]。
D+1 表示 非可逆 FQL 可用

核心映射

r = ||xyz||
u = xyz / max(r, eps)
decomposed = [u, log(r), other]

xyz = normalize(u) * exp(log_r)
action = clip([xyz, other], -1, 1)

实现方式

  • actor base distribution 仍是 D 维 Gaussian，但前三维 latent 解释为 theta_raw, phi_raw, speed_raw。
  • bounded sigmoid 将 theta 映射到 (0,π)，phi 映射到 (0,2π)，speed 映射到 (0,max_speed)。
  • log-det 包含球坐标体积项 r² sin(theta) 和三条 sigmoid 的导数。
可逆 bijector Jacobian-corrected 极点附近更敏感

核心映射

theta = π * sigmoid(theta_raw)
phi = 2π * sigmoid(phi_raw)
r = max_speed * sigmoid(speed_raw)

x = r sin(theta) cos(phi)
y = r sin(theta) sin(phi)
z = r cos(theta)

实现方式

  • 前三维 latent 是 plane_x, plane_y, speed_raw，先在平面采样，再 inverse stereographic 到单位球。
  • 方向映射没有球坐标角度极点，实验中作为 RLPD/SAC 的 DS 主线。
  • log-det 使用球极投影面积项 4 / (1 + rho²)²，再乘上 r² 和 speed sigmoid 导数。
推荐主线 可逆 bijector 方差更稳

核心映射

rho2 = px² + py²
direction = [
  2px / (1 + rho2),
  2py / (1 + rho2),
  (1 - rho2) / (1 + rho2)
]

action_xyz = speed * direction
